# Architecture Requirements Baseline (Planning Only)

更新日: 2026-03-16

## 1. 目的とスコープ
- 本ドキュメントは、アーキテクチャ設計フェーズ向けの要件記録（planning）を目的とする。
- 実装方針・コード変更・運用手順変更は含めない。
- 対象は、履歴書取り込みから候補者検索・評価・説明生成までの全体システム。


## 3. 要件一覧（初期ベースライン）

### 3.1 ビジネス要件（BR）
- `BR-01` 大量履歴書から、自然言語の求人要件に対する高関連候補を短時間で提示できること。
- `BR-02` 単純キーワード一致ではなく、文脈・転用可能スキル・キャリア軌跡を考慮すること。
- `BR-03` 推薦理由を説明可能な形で提示し、採用担当者の意思決定を支援すること。

### 3.2 機能要件（FR）
- `FR-01` ユーザークエリを元に関連する候補者をRAGベースで検索すること。
- `FR-01-01` ユーザークエリからスキル、職種、業種、経験レベル、学歴をハードフィルタ用に抽出すること。 
- `FR-01-02` ユーザークエリから抽出したフィルタ条件とAPIで受け付けるフィルタ条件に齟齬がある場合は、ユーザーにリトライを促すこと。
- `FR-01-03` ベクトル検索前にスキル、職種、業種、経験レベル、学歴でハードフィルタをかけて候補者集合を絞り込むこと。
- `FR-01-03-01` クエリから抽出したスキル語・職種語を ESCO ID に正規化する query normalizer を実装すること。
- `FR-01-03-02` query normalizer は少なくとも `exact -> alt_label -> fuzzy -> embedding` の順で ESCO マッチングを試行できること。
- `FR-01-03-03` query normalizer は単一IDではなく、confidence 付きの ESCO 候補集合を返却できること。
- `FR-01-03-04` high confidence の ESCO 候補は `skill_esco_ids_json` および `occupation_esco_ids_json` に対する hard filter に利用できること。
- `FR-01-03-05` medium confidence の ESCO 候補は hard filter ではなく rerank feature として利用できること。
- `FR-01-03-06` low confidence の ESCO 候補は hard filter に利用せず、vector query 側の自由語彙としてのみ利用すること。
- `FR-01-03-07` 業種、経験レベル、学歴の hard filter は、Milvus scalar metadata に対して適用できること。
- `FR-01-04` ユーザークエリを元にスキルのVectorDBに対してベクトル検索を行い、関連候補者を取得すること。
- `FR-01-04-01` スキルの関連度スコアを計算すること。
- `FR-01-04-02` ユーザークエリからスキルのベクトル検索用のクエリを生成すること。
- `FR-01-05` ユーザークエリを元に職種のVectorDBに対してベクトル検索を行い、関連候補者を取得すること。
- `FR-01-05-01` 職種の関連度スコアを計算すること。
- `FR-01-05-02` ユーザークエリから職種のベクトル検索用のクエリを生成すること。


- `FR-02` クエリごとにキーワード検索とベクトル検索を並列実行する。
- `FR-02-01` 両検索結果を重複排除して統合し、RRF（または重み付き和）で融合スコアを算出する。
- `FR-02-02` 統合候補集合（Top-K）をCross-encoder再ランキング入力として出力する。
- `FR-02-03` 各候補に keyword_score vector_score fusion_score を保持し、説明可能性に利用できること。

- `FR-03`  `FR-02-02` の候補集合に対して、Cross-encoderでクエリと候補者情報の関連度スコアを算出し、再ランキングすること。
- `FR-03-01` 最終再ランキングスコアは `fusion_score` と `cross_encoder_score` の重み付き統合で算出すること。
- `FR-03-02` 再ランキング後の上位N件のみを後段スコアリング（`FR-04`）入力に渡すこと。
- `FR-03-03` Cross-encoderが利用不可の場合は `FR-02-01` の融合順位を用いて処理継続すること（fallback）。



- `FR-04` RetrieveしRe-rankingした候補者をスキルベースのセマンティックマッチングでスコアリングすること。 
- `FR-04-01` スキルがどの程度求人要件と一致するかを評価すること
- `FR-04-01-01` 経験やサマリからスキルがどのような文脈で使われているかを考慮してスキルの熟練度とユーザークエリとのマッチング度をAI　Agentを用いて評価すること（Few-shotプロンプトが望ましい） 
- `FR-04-02` 経験やサマリから転用可能スキルを挙げて、熟練度とユーザークエリとのマッチング度を計算し、どの程度求人要件と一致するかをAI　Agentを用いて評価すること
- `FR-04-03` 経験がどの程度求人要件と一致するかを経験年数と経験した仕事内容からAIAgentも用いて評価すること
- `FR-04-03-01` ユーザークエリで指定された経験が候補者の経験の中でどれくらい最近かをAIAgentを用いて評価すること（最近であればあるほど高いスコアとする）
- `FR-04-04` 職種がどの程度求人要件と一致するかをAIAgentを用いて評価すること
- `FR-04-05` 学歴がどの程度求人要件と一致するかをAIAgentを用いて評価すること
- `FR-04-06` 資格がどの程度求人要件と一致するかをAIAgentを用いて評価すること
- `FR-04-07` キャリアprogression分析として、特定スキルの成長速度（縦方向の成長）と責任範囲の拡大スピード（横方向の成長）をAIAgentを用いて評価すること
- `FR-04-08` コミュニケーションやソフトスキルの指標をAIAgentを用いて評価すること
- `FR-04-09` FR-04-01～FR-04-08の評価結果を総合して、総合したスキルマッチングスコアを算出し、最終ランキングスコアに統合すること

- `FR-05` ランキング順に候補者を返し、採用担当者が推薦理由を理解できるように、スキルマッチングの内訳（どのスキルが求人要件とどの程度一致しているか）を説明可能な形で提示すること。
- `FR-05-01` 候補者一覧は最終ランキングスコアの降順で返却すること。
- `FR-05-02` 各候補者について、総合マッチングスコア、順位、推薦理由サマリを返却すること。
- `FR-05-03` 各候補者について、求人要件ごとのスキル一致内訳を返却すること。
- `FR-05-03-01` スキル一致内訳には、求人要件スキル名、候補者保有スキル名、一致度スコアを含めること。
- `FR-05-03-02` スキル一致内訳には、一致種別（完全一致、意味類似一致、転用可能スキル一致）を含めること。
- `FR-05-03-03` スキル一致内訳には、根拠となる職務経験、サマリ、資格、学歴などの参照元情報を含めること。
- `FR-05-04` 各候補者について、強みとなる一致項目と不足している主要スキルを区別して提示すること。
- `FR-05-05` 推薦理由はAPI返却可能な構造化データとして保持し、UI上で人が読める説明文としても提示できること。
- `FR-05-06` 説明生成に必要な詳細情報が不足する場合でも、保持済みスコアと根拠データを用いたテンプレートベース説明で返却を継続できること。
- `FR-05-07` 各候補者の履歴書の生データをAPIで取得できるようにすること（説明生成のための追加情報取得や、採用担当者が直接内容を確認するため）。

- `FR-06` コア機能をAPIエンドポイントとして提供すること。
- `FR-06-01` 自然言語の求人要件と検索条件を受け取り、候補者検索を実行するAPIを提供すること。
- `FR-06-01-01` 検索APIは、職種、業種、スキル、経験レベル、学歴、勤務地などのフィルタ条件を受け付けること。
- `FR-06-01-02` 検索APIは、ハイブリッド検索、再ランキング、最終スコアリングを経た候補者一覧を返却すること。
- `FR-06-02` 検索結果として、候補者ID、順位、総合スコア、推薦理由サマリを返却すること。
- `FR-06-02-01` 検索結果として、スキル一致内訳、経験一致内訳、主要な不足項目を返却できること。
- `FR-06-03` 候補者単位の詳細情報を取得するAPIを提供すること。
- `FR-06-03-01` 候補者詳細APIは、構造化プロフィール、経験、学歴、スキル、説明情報を返却すること。
- `FR-06-04` 説明可能なランキング根拠を取得するAPIを提供すること。
- `FR-06-04-01` ランキング根拠APIは、各候補者のスコア内訳と根拠データを返却すること。
- `FR-06-05` APIの入出力は、他システムが利用可能な構造化JSON形式で提供すること。

- `FR-07` 各入出力にガードレールを実装すること
- `FR-07-01` 求人要件入力のガードレール（妥当性チェック）を実装すること。
- `FR-07-01-01` 求人要件が自然言語で記述されていることを確認すること。
- `FR-07-01-02` 求人要件に必要な情報（職種、スキル、経験レベル、学歴など）が含まれているかをチェックすること。
- `FR-07-01-03` 求人要件が過度に短すぎたり、長すぎたりしないようにすること。
- `FR-07-01-04` 求人要件に不適切な内容（差別的表現、個人情報など）が含まれていないかをチェックすること。
- `FR-07-01-05` 求人要件がガードレールに違反している場合は、エラーメッセージを返し、修正を促すこと。
- `FR-07-02` パイプラインの方で履歴書パースのバリデーションを実装すること。
- `FR-07-02-01` 履歴書の必須セクション（職務経験、学歴、スキルなど）が正しく抽出されているかを確認すること。
- `FR-07-02-02` 抽出された情報が正確で一貫性があるかをチェックすること。
- `FR-07-03` 最終出力される候補者情報に対して、経験レベル、職種カテゴリ、業種、学歴などのメタデータフィルタを実装すること。
- `FR-07-03-01` メタデータフィルタは、ユーザークエリで指定された条件に基づいて候補者をフィルタリングすること。
- `FR-07-03-02` メタデータフィルタは、検索APIの入力パラメータとしても受け付けること。
- `FR-07-03-03` 業種フィルタは、`normalized_candidates` に格納された ESCO 職種階層の親コンポーネントを業種として扱い、その値に基づいて候補者をフィルタリングできること。
- `FR-07-04` 検索結果返却前に、ランキング結果と説明文に対して禁止属性および代理変数の利用を監査すること。
- `FR-07-04-01` 監査では、性別、年齢、国籍、人種、宗教、婚姻状況などの禁止属性と、それらを示唆する代理変数が順位付け根拠や説明文に含まれていないことを確認すること。
- `FR-07-04-02` 監査で問題が見つかった場合は、警告を返却し、説明文を安全なテンプレート表現に置換し、レビュー可能な監査ログを記録すること。

- `FR-08` DeepEvalやカスタム指標（スキルカバレッジ、経験適合度、カルチャーマッチ）を用いたマッチング品質評価を実装すること。
- `FR-08-01` 評価用データセットとして、求人要件、候補者集合、期待順位または正解ラベルを保持できること。
- `FR-08-02` DeepEvalを用いて、検索結果および推薦理由の品質を評価できること。
- `FR-08-02-01` 評価は、上位候補集合に対する多様性メトリクスを算出できること。
- `FR-08-02-02` 多様性メトリクスは、少なくとも職種、業種、経験レベルの偏りを検出できること。
- `FR-08-03` カスタム指標として、求人要件に対するスキルカバレッジを算出できること。
- `FR-08-03-01` スキルカバレッジは、必須スキルと推奨スキルを区別して算出できること。
- `FR-08-04` カスタム指標として、候補者の経験適合度を算出できること。
- `FR-08-04-01` 経験適合度は、経験年数、経験内容、直近性を考慮して算出できること。
- `FR-08-05` カスタム指標として、カルチャーマッチまたはソフトスキル適合度を算出できること。
- `FR-08-05-01` カルチャーマッチ評価は、履歴書中の行動特性、コミュニケーション指標、チーム協働に関する記述を根拠として算出できること。
- `FR-08-06` 評価は、ベースライン手法と改善手法（ハイブリッド検索、Cross-encoder再ランキング、Agent評価）を比較できること。
- `FR-08-07` 評価結果は、指標ごとの集計値とサンプル単位の詳細結果をレポートとして出力できること。
- `FR-08-08` 評価結果に基づき、所定の品質閾値を満たすか判定できること。

- `FR-09` 採用担当者向けのシンプルなフロントエンド画面を提供すること。
- `FR-09-01` 初期画面では、自然言語の求人要件を入力できるチャットボックスを提供すること。
- `FR-09-02` 初期画面では、職種、業種、スキル、経験レベル、学歴などのフィルタ条件を選択できる UI を提供すること。
- `FR-09-03` 検索結果画面では、候補者情報を横長カード形式でランキング順に表示すること。
- `FR-09-03-01` 候補者カードには、候補者IDまたは表示名、総合スコア、主要スキル、推薦理由サマリを表示すること。
- `FR-09-04` 候補者カードをクリックした際に詳細画面へ遷移できること。
- `FR-09-04-01` 詳細画面では、構造化プロフィール、経験、学歴、スキル、説明情報、履歴書の生データを確認できること。

### 3.3 パイプライン要件（PR）
- `PR-01` 取り込みフェーズでは、履歴書ソースと ESCO ソースを取り込み、MongoDB に永続化すること。
- `PR-01-01` 履歴書ソースは `source_dataset` と `source_record_id` により一意に識別され、重複投入時も冪等に更新できること。
- `PR-01-02` ESCO ソースは後続の正規化と埋め込み生成で利用できる形で `raw_esco_*` として保持すること。
- `PR-02` ESCO embedding フェーズでは、`raw_esco_*` を埋め込み化し、Milvus などの VectorDB に検索可能な形で格納すること。
- `PR-02-01` occupation embedding には preferred label、alt label、description、hierarchy、essential skills を含められること。
- `PR-02-02` skill embedding には preferred label、alt label、description、hierarchy、related occupations を含められること。
- `PR-03` parsing / extraction フェーズでは、`source_1st_resumes` から parsed sections と extracted fields を生成すること。
- `PR-03-01` parsed sections はセクション種別、タイトル、本文、文字数を保持できること。
- `PR-03-02` extracted fields は少なくとも経験、学歴、スキル、職種候補、抽出方式、抽出バージョンを保持できること。
- `PR-04` normalized フェーズでは、`source_1st_resumes`、`raw_esco_*`、VectorDB を用いて `normalized_candidates` を生成すること。
- `PR-04-01` 正規化では lexical candidate generation、embedding retrieval、LLM candidate generation を統合できること。
- `PR-04-02` 正規化結果には occupation candidates、skill candidates、experience / education、`llm_handoff`、`matching_debug` を保持すること。
- `PR-04-03` embedding または LLM が利用不可の場合でも、lexical ベースの fallback で処理継続できること。
- `PR-05` normalized eval フェーズでは、正規化結果に対する品質評価を実行し、Markdown または JSON 形式のレポートを出力できること。
- `PR-05-01` 評価フェーズは、少なくとも正規化評価と LLM 評価を実施できること。
- `PR-06` real embedding フェーズでは、`normalized_candidates` から RAG / semantic search 用の候補者ベクトルを生成できること。
- `PR-06-01` 候補者ベクトルは、検索サービス向けに publish 可能なスナップショットとして管理できること。

### 3.4 後続スコープ（Phase 2）
- `P2-01` 採用担当者のフィードバックループによるランキングモデル改善。
- `P2-02` スキル需要や候補者傾向を示す採用分析ダッシュボード。
- `P2-03` 面接日程調整エージェントへのエージェントハンドオフ。
- `P2-04` 採用担当者と Hiring Manager 間の Agent-to-Agent（A2A）コミュニケーション。
